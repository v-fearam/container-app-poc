¿Por qué necesitamos NDv2?

El sistema actual no puede escalar

Una sola base de datos para todo: eventos, configuración, Hangfire, consultas. Sin depuración programada. Sin escalabilidad horizontal.

🔥 Competencia de Recursos

2 instancias de Hangfire + API + recolección de eventos, todos compitiendo por las mismas tablas inmensas.

🚫 Imposible Escalar

El cuello de botella es la base de datos compartida. Agregar más workers no ayuda.

🙈 Sin Visibilidad

Dashboard de Hangfire muestra jobs, no el estado del negocio. No sabemos qué está fallando.

💣 Bomba de Tiempo

Se promete más funcionalidad (Huemul, nuevos procesos) sobre un sistema que ya no da más.

Antes vs Después

❌ NDv1 (Estado Actual)

  • Una sola base de datos SQL para todo
  • Hangfire con infraestructura always-on
  • Sin depuración programada (años de datos)
  • Sin escalabilidad horizontal
  • Visibilidad limitada (logs de Hangfire)
  • VMs (IaaS) — mantener OS, parches, actualizaciones
  • Cuello de botella en la BD compartida

✅ NDv2 (Nueva Arquitectura)

  • Bases separadas: Cosmos (ejecución), SQL (consumo), Dashboard (config)
  • Container Apps Jobs (efímero, $0 cuando no corre)
  • TTL automático en Cosmos (45 días), particiones en SQL (4 años)
  • KEDA autoscaling (0 → 50 workers dinámico)
  • Dashboard operacional en tiempo real
  • 100% PaaS — Azure gestiona infraestructura, OS, scheduler
  • Escala horizontalmente sin límites

Ventajas de la Nueva Solución

📈 Escalabilidad Real

Workers escalan de 0 a 50 automáticamente con KEDA. CosmosDB y SQL manejan millones de docs/mes sin problemas.

💰 Costos Optimizados

Container Apps Jobs solo pagan por ejecución (segundos). Workers en 0 replicas = $0. Sin infraestructura always-on.

🔍 Visibilidad Total

Dashboard operacional con métricas en tiempo real: comunicaciones procesadas, estados, errores, DLQ, workers activos.

🛡️ Resiliente

Dead Letter Queues automáticas, retry policies, Change Feed garantiza sincronización, ARM API para gestionar jobs.

🗑️ Depuración Automática

TTL 45 días en CosmosDB. Eventos crudos en Blob: Hot 30d → Cold 180d → Archive 4 años → Delete. Particiones por año en SQL con retención de 4 años.

🔐 Seguridad

Managed Identity en todo (sin secrets), Key Vault para configuración, Private Endpoints, Easy Auth con Entra ID.

⚡ Event-Driven

Change Feed sincroniza automáticamente. Service Bus desacopla componentes. No hay polling ni queries pesados.

🎯 Separación por Vertical

Genéricos, Negocio y Campañas son verticales independientes: cada una con su modelo estrella, queues, workers y config. Escalan por separado.

POC: Patrones Validados

✅ Easy Auth + Entra ID

Platform-managed auth. Frontend y backend con SSO. Token store habilitado. Cross-tenant validado.

✅ Workers + KEDA

WeatherWorker y DashboardWorker escalan con Service Bus queue depth. Graceful shutdown implementado.

✅ Change Feed → SQL

Sincronización automática CosmosDB → SQL Server. Idempotencia con timestamps. Lease-based scaling.

✅ Container Jobs (Scheduled)

WeatherEnqueuer ejecuta cada 5 min vía CRON. Ephemeral execution. Logs en Log Analytics.

✅ Dashboard Operacional

React + shadcn/ui. Infrastructure Health API. Gestión de jobs (trigger, stop, schedule). Métricas en tiempo real.

✅ Managed Identity Everywhere

Backend, frontend, workers y jobs con MI. RBAC en Bicep. SQL con Active Directory Default. Sin secrets en código.

✅ Key Vault References

App Insights connection string, auth client secrets. Auto-seeded en Bicep. RBAC-enabled (no access policies).

✅ Dead Letter Queue

DLQ Manager UI: peek, re-queue, discard. Logs con DeadLetterReason. Azure SDK con Moq (sin wrappers).
